# 0.1.4 — 2026-07-11

## Added

- **`postalCode`** on `npi_search_providers` output rows and the `format()` location line, sourced from the NPPES LOCATION address. ([#4](https://github.com/cyanheads/npi-providers-mcp-server/issues/4))
- **Supply-chain guards** — `bunfig.toml` blocks package versions published less than 3 days ago (`install.minimumReleaseAge`, excluding `@cyanheads/mcp-ts-core`, adopted same-day) and scans installs with `@socketsecurity/bun-security-scanner`.
- `.gitattributes`, `.github/FUNDING.yml`, `.github/SECURITY.md`.

## Changed

- `Dockerfile` pins `oven/bun:1.3.14`, adds BuildKit cache mounts on the install layers, and builds with `--ignore-scripts`.
- `devcheck.ts`'s outdated-package check no longer flags a row held back solely by `minimumReleaseAge` as a finding.

## Fixed

- **`npi_search_providers` location filtering** — NPPES does not treat `city`/`state`/`postal_code` as a hard filter for specialty searches; matching rows are now post-filtered server-side to the requested location, with the dropped-row count disclosed via `ctx.enrich.notice`. Truncation keys on the raw upstream page size so post-filtering can't hide a full page. ([#4](https://github.com/cyanheads/npi-providers-mcp-server/issues/4))
- **`state` input** accepts a blank string (`""`) from form-based MCP clients — treated as omitted rather than a validation failure; the 2-letter regex is still enforced for non-empty values. ([#5](https://github.com/cyanheads/npi-providers-mcp-server/issues/5))
